# Case patterns

Recurring product-decision patterns, anonymized. Each one captures a kind of mistake (or near-mistake) that shows up across many companies, sectors, and products.

The point of these is not to memorize the cases. It's to recognize the *shape* of the problem when you encounter it in the wild — and to know which framework applies.

| Pattern | The shape of the problem |
|---|---|
| [Wrong-bottleneck diagnosis](wrong-bottleneck.md) | Team solves a real, validated user pain — but it's not the constraint that determines user behavior. |
| [PMF without business-model fit](pmf-without-business-fit.md) | Users love the product. Unit economics, regulation, or partner constraints make it impossible to scale. |
| [Refactor vs. rebuild](refactor-vs-rebuild.md) | Legacy product, eroding customer trust, demoralized team. Four options, each with different reversibility profiles. |
| [Hardware as a one-way door](hardware-irreversible.md) | Static mockups missed an issue that only appeared in motion / in real conditions. The cost of the miss is structural. |
| [Iterative experimentation under crisis](iterative-experimentation.md) | Pre-defined success criteria + fast iteration turned an existential threat into a multi-billion-dollar new revenue line. |
| [Mini cases: PM/engineering tension](mini-cases-engineering-pm.md) | Two short patterns: stated user desires vs. technical constraints, and product readiness vs. GTM readiness. |
